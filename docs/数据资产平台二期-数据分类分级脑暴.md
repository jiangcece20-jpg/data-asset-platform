# 数据分类分级与权限申请审批 — 脑暴速写

## 1. 核心目标与动机

### 要解决的问题
- 平台已有多种数据资产类型（数据表、API、报表、指标、标签），用户想使用这些资产时，缺少统一的权限申请与审批机制
- 不同类型、不同业务线的数据资产，审批流程和审批人不同，需要灵活的审批路由能力
- 用户经常需要同时申请多个资产的权限，缺少批量申请能力
- 当前平台的权限申请只做到了"申请中"的 mock 状态，没有真实审批流

### 核心目标
- 建立基于"资产类型 × 业务线 × 业务负责人 × 来源平台"的审批路由引擎
- 对接飞书审批流，实现真实的审批创建、流转、回调
- 支持用户批量提交权限申请，后台按审批路由规则自动拆单
- 管理员可在后台配置审批模板，灵活适配不同业务场景

## 2. 目标用户画像

### 普通数据用户（申请人）
- 在资源发现/资产目录中找到需要的数据资产
- 发起权限申请（单个或批量）
- 跟踪申请进度，查看审批结果
- 典型场景：数据分析师需要查询某张表、运营人员需要查看某个报表

### 审批人
- 在飞书中收到审批通知
- 审批通过或驳回（需填写驳回原因）
- 审批人角色包括：申请人上级、资产负责人、业务线审批人
- 注意：审批动作发生在飞书侧，不在本平台内

### 管理员
- 配置审批模板（审批路由规则）
- 管理审批流与飞书审批定义的映射关系
- 查看审批统计与异常处理

## 3. 使用场景与用户故事

### 场景 1：单个资产申请权限
用户在详情页点击"申请权限"→ 填写申请理由 → 提交 → 平台根据资产属性匹配审批模板 → 调用飞书 API 创建审批实例 → 审批人在飞书中审批 → 飞书 webhook 回调平台 → 平台更新权限状态

### 场景 2：批量申请权限
用户在资源发现页勾选多个资产 → 点击"批量申请权限"→ 填写统一的申请理由 → 提交 → 平台按审批路由规则自动拆成 N 个审批单 → 每个审批单独立走飞书审批 → 用户在"我的申请"中看到 1 条主申请，点进去看 N 个子单进度

### 场景 3：部分驳回
用户批量申请了 5 个资产 → 3 个通过、1 个驳回、1 个审批中 → 用户在主申请中看到整体状态为"部分通过"→ 已通过的 3 个立即生效 → 驳回的 1 个可以查看驳回原因并重新申请 → 审批中的继续等待

### 场景 4：报表按业务线审批
用户申请金融业务线的报表 → 平台匹配到"报表 + 金融"审批模板 → 审批流走金融业务线审批人 → 审批通过后用户获得该报表权限

### 场景 5：数据表串行审批
用户申请某张数据表 → 平台匹配到"数据表"审批模板 → 第一步：申请人上级审批（通过组织架构接口获取）→ 上级通过后 → 第二步：表负责人审批（可能多个，走会签）→ 全部通过后用户获得权限

### 场景 6：管理员配置审批模板
管理员进入审批管理后台 → 创建审批模板 → 设置匹配条件（资产类型 + 业务线 + 来源平台）→ 设置审批节点链（串行，每个节点指定审批人来源）→ 关联飞书审批定义 → 保存启用

## 4. 关键设计决策（已确认）

| 决策项 | 结论 |
|---|---|
| 审批系统 | 对接飞书审批（首次对接） |
| 审批创建方式 | 通过飞书 API 创建审批实例 |
| 审批结果回调 | 优先 webhook 回调，备选轮询（看技术方案） |
| 审批流模型 | 串行审批（上级审批完 → 负责人审批） |
| 审批模板配置粒度 | 资产类型 × 业务线 × 业务负责人 × 来源平台 |
| 审批模板管理 | 管理员后台配置 |
| 批量申请 | 支持，后台按审批路由自动拆单 |
| 拆单后展示 | 1 条主申请 + N 个子单（方案 A） |
| 申请时填写内容 | 申请理由 |
| 申请人上级获取 | 组织架构接口 |
| 权限粒度 | 资产级别（整张表/整个报表） |
| 权限有效期 | 无有效期，永久 |
| 分类维度 | "类型"和"业务线"是独立维度 |

## 5. 需要深入讨论的问题

### 5.1 部分驳回的处理策略

批量申请拆单后，各子单独立审批，必然出现"部分通过、部分驳回"的情况。

**主申请状态机设计建议：**

```
审批中 → 部分通过（有子单通过，有子单仍在审批或已驳回）
       → 全部通过
       → 全部驳回
       → 已撤回（用户主动撤回未完成的子单）
```

**部分驳回时的处理规则：**
- 已通过的子单：权限立即生效，不受其他子单影响
- 已驳回的子单：展示驳回原因，支持"重新申请"（只重新提交被驳回的子单）
- 审批中的子单：继续等待，不受已驳回子单影响
- 主申请状态：只有当所有子单都有终态时，主申请才进入终态

**用户侧展示建议：**
- 主申请卡片显示进度条或比例：`3/5 已通过 · 1/5 已驳回 · 1/5 审批中`
- 支持按子单状态筛选
- 驳回子单旁边有"重新申请"按钮

### 5.2 多负责人审批（会签 vs 或签）

数据表场景下，表可能有多个负责人。需要确认：
- **会签（全部同意才通过）**：更严格，适合高安全等级数据
- **或签（任一同意即通过）**：更高效，适合一般数据

**结论：** 负责人审批节点走或签（任一负责人同意即通过），降低审批阻塞风险。

### 5.3 审批模板匹配优先级

当一个资产同时命中多个模板时，如何选择？

**建议匹配优先级（从高到低）：**
1. 资产类型 + 业务线 + 来源平台 + 业务负责人（精确匹配）
2. 资产类型 + 业务线 + 来源平台（忽略负责人）
3. 资产类型 + 业务线（忽略来源平台）
4. 资产类型（仅类型匹配）
5. 默认模板（兜底）

管理员配置时需要能看到优先级顺序，避免规则冲突。

### 5.4 飞书审批对接的技术边界

需要调研确认：
- 飞书审批定义（审批模板）是在飞书后台创建，还是通过 API 动态创建？
- 飞书是否支持串行多节点审批？（支持，飞书审批定义中可配置多个审批节点）
- 飞书是否支持会签节点？（支持）
- 飞书 webhook 回调的可靠性如何？是否需要补偿机制（定时轮询兜底）？
- 组织架构接口是飞书的还是公司内部的？

### 5.5 审批模板与飞书审批定义的关系

两种方案：
- **方案 A：一对一映射** — 平台每个审批模板对应飞书一个审批定义。管理员在飞书后台也要同步创建。维护成本高但控制力强。
- **方案 B：共用少量飞书审批定义** — 平台用少量通用飞书审批定义（比如"数据权限申请-串行两级"），通过 API 传参动态指定审批人。维护成本低但灵活性受限于飞书审批定义的表单结构。

**建议方案 B**，用 2-3 个飞书审批定义覆盖所有场景：
1. 单级审批（只需负责人审批）
2. 两级串行审批（上级 → 负责人）
3. 两级串行 + 会签审批（上级 → 多负责人会签）

### 5.6 "其它业务线"的兜底策略

金融和物流已明确审批流，其它业务线暂未确定。

**建议：**
- 配置一个"默认审批模板"作为兜底
- 默认模板走"资产负责人审批"（最简单的单级审批）
- 管理员后续可以随时为新业务线添加专属模板
- 新模板生效后，该业务线的申请自动走新模板，无需迁移历史数据

### 5.7 与现有权限体系的衔接

当前平台详情页已有"申请权限"按钮和三种状态（待申请/申请中/已有权限）。

需要确认：
- 审批通过后，权限是自动下发到数据源（比如 Hive 授权），还是只在平台层面标记？
- 如果只是平台标记，后续真实权限下发怎么做？
- 是否需要"权限回收"能力？（管理员主动回收、离职自动回收等）

## 6. 灵感参考与竞品

### 业界常见做法
- **阿里 DataWorks**：数据分类分级 + 审批流，按安全等级自动匹配审批链路
- **华为 DataArts**：数据安全中心，分类分级标签 + 权限策略联动
- **字节 DataLeap**：飞书审批深度集成，支持自定义审批流模板
- **AWS Lake Formation**：基于标签的访问控制（Tag-Based Access Control）

### 可借鉴的设计模式
- 审批路由引擎：条件匹配 → 模板选择 → 节点链生成
- 批量拆单 + 主子单模式：电商订单拆单的成熟模式
- 审批模板可视化配置：拖拽式节点编排（但 MVP 可以先用表单配置）

## 7. 初步功能模块划分

### 用户侧
- 权限申请入口（详情页 + 资源发现页批量）
- 申请表单（申请理由 + 资产列表确认）
- 我的申请列表（主申请 + 子单详情）
- 申请状态跟踪（审批中/部分通过/全部通过/全部驳回/已撤回）
- 重新申请（针对被驳回的子单）

### 管理员侧
- 审批模板管理（CRUD + 优先级排序）
- 审批模板配置（匹配条件 + 审批节点链 + 飞书审批定义关联）
- 审批记录查看（全局视角）
- 审批统计（通过率、平均审批时长等，后续可做）

### 后台引擎
- 审批路由引擎（资产属性 → 模板匹配 → 审批链路生成）
- 批量拆单引擎（按审批模板分组拆单）
- 飞书审批对接层（创建实例、webhook 回调处理、状态同步）
- 组织架构对接层（获取申请人上级）

## 8. MVP 范围建议

### P0（必须做）
- 单个资产权限申请 + 飞书审批对接
- 审批模板管理（管理员配置）
- 审批路由引擎（按资产属性匹配模板）
- 串行审批（上级 → 负责人）
- 我的申请列表 + 状态跟踪
- 飞书 webhook 回调处理

### P1（批量能力）
- 批量申请 + 自动拆单
- 主子单模式展示
- 部分驳回处理 + 重新申请
- 会签/或签配置

### P2（增强能力）
- 审批统计看板
- 审批催办
- 权限回收
- 审批超时自动处理
- 安全等级与审批链路自动关联
