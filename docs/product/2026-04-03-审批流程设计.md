# 数据资产管理平台 - 审批流程设计

## 1. 文档信息

- 文档版本：V1.0
- 文档日期：2026-04-03
- 文档状态：设计稿
- 适用范围：资源上架审批、权限审批、负责人交接

---

## 2. 背景

当前原型中所有操作（上架、下架、权限申请等）均直接生效，无审批环节。资源的上下架不影响数据的实际使用，仅作为数据质量的标记——上架代表"质量达标，推荐使用"，下架代表"质量存疑，谨慎使用"。

在此前提下，审批的目的不是风控，而是**保证资产目录的含金量**和**数据安全的基本门槛**。

---

## 3. 审批流程定义

### 3.1 三种审批流程

| 流程 | 触发条件 | 审批人 | 审批内容 | 结果 |
|------|---------|--------|---------|------|
| 上架审批 | 负责人提交上架申请 | 业务负责人 | 业务语义准确性、目录合理性 | 通过→已上架 / 驳回→待维护 |
| 权限审批 | 用户申请数据权限 | 资源技术负责人 | 需求合理性 | 通过→已有权限 / 拒绝→回到待申请 |
| 负责人交接 | 当前负责人发起交接 | 接收方（目标负责人） | 是否接手 | 确认→生效 / 拒绝→不变 |

### 3.2 不需要审批的操作

| 操作 | 说明 |
|------|------|
| 下架 | 直接生效，不影响数据使用，仅撤回质量标记 |
| 标记不上架 | 内部治理决策，不影响外部 |
| 转待维护 | 内部状态回转 |
| 编辑描述类字段 | 字段描述、表描述、中文名、使用说明，即时生效 |
| 收藏/取消收藏 | 个人操作 |

## 4. 状态流转

### 4.1 资源状态定义

| 状态 | 标识 | 说明 |
|------|------|------|
| 待维护 | `maintain` | 采集进入系统后的初始状态，需补充信息 |
| 待审核 | `reviewing` | 已提交上架申请，等待审批 |
| 已上架 | `published` | 审批通过，资源成为正式资产 |
| 不上架 | `no-list` | 标记为不需要上架 |

### 4.2 状态流转图

```
                     ┌───────────────────────────────────────┐
                     │                                       │
                     ▼                                       │
   ┌──────────┐  提交上架  ┌──────────┐  审核通过  ┌──────────┐
   │  待维护   │─────────→│  待审核   │─────────→│  已上架   │
   │ maintain  │  (审批)   │reviewing │          │published │
   └──────────┘           └──────────┘          └──────────┘
       ▲   │                   │                     │
       │   │ 不上架             │ 驳回(必填原因)        │ 下架
       │   │ (直接)            │                     │ (直接)
       │   ▼                   │                     │
   ┌──────────┐               │                     │
   │  不上架   │←──────────────┘                     │
   │  no-list │  转待维护(直接)                       │
   └──────────┘←──────────────────────────────────────┘
       │                                       
       └──── 转待维护(直接) ──→ 待维护

   特殊路径：已上架 → 修改目录 → 待审核（重新审批）
```

## 5. 编辑规则

| 字段类别 | 可编辑状态 | 生效方式 |
|---------|-----------|---------|
| 描述类（字段描述、表描述、中文名、使用说明、摘要） | 任何状态 | 即时生效 |
| 目录 | 任何状态 | 未上架：即时生效；已上架：状态变为"待审核"，需重新审批 |
| 负责人 | 任何状态 | 接收方确认后生效 |

## 6. 上架审批详细流程

### 6.1 前置校验

提交上架前，系统自动校验以下必填项，不满足则无法提交：

| 校验项 | 检查规则 |
|--------|---------|
| 中文名（display） | 非空 |
| 摘要（summary） | 非空 |
| 目录（catalog） | 非空 |
| 负责人（tech_owner） | 非空 |

### 6.2 审批人判断内容

| 判断项 | 说明 |
|--------|------|
| 业务语义准确性 | 中文名、摘要是否能让人看懂 |
| 目录合理性 | 是否挂到正确的目录节点下 |

### 6.3 流程步骤

1. 负责人在资源管理中点击"提交上架"
2. 系统校验必填项，通过后弹出确认窗口展示校验结果
3. 确认提交后，资源状态变为"待审核"
4. 审批人（业务负责人）在"待审核"Tab 中看到申请
5. 审批人审核后选择"通过"或"驳回"
6. 通过：资源状态变为"已上架"
7. 驳回：必须填写驳回原因，资源状态回到"待维护"

### 6.4 撤回机制

资源处于"待审核"状态时，提交人可以主动撤回申请，状态回到"待维护"。

### 6.5 已上架资源修改目录

已上架资源修改目录后，状态自动变为"待审核"，需要重新走上架审批流程。

---

## 7. 权限审批详细流程

### 7.1 流程步骤

1. 用户在详情页点击"申请权限"
2. 填写需求说明，提交申请
3. 权限状态变为"申请中"
4. 资源技术负责人在权限中心看到申请
5. 审批人选择"通过"或"拒绝"
6. 通过：用户获得权限（可设有效期）
7. 拒绝：必须填写拒绝原因，用户可重新申请

### 7.2 权限状态

| 状态 | 标识 | 说明 |
|------|------|------|
| 待申请 | `none` | 未申请，显示"申请权限"按钮 |
| 申请中 | `applying` | 已提交，等待审批 |
| 已有权限 | `granted` | 审批通过 |

---

## 8. 负责人交接详细流程

### 8.1 流程步骤

1. 当前负责人发起交接，选择接收方并填写原因
2. 系统创建交接请求，状态为"待确认"
3. 接收方在工作台待办中看到交接请求
4. 接收方选择"确认接手"或"拒绝"
5. 确认：负责人变更即时生效
6. 拒绝：交接取消，通知原发起人

---

## 9. 资源管理页面操作矩阵

### 9.1 各状态下的单资源操作

| 状态 | 可用操作 | 操作类型 |
|------|---------|---------|
| 待维护 | 编辑、修改目录、提交上架、标记不上架、标签、交接 | 提交上架需审批，其余直接生效 |
| 待审核 | 撤回 | 直接生效 |
| 已上架 | 下架、标签、交接 | 直接生效 |
| 不上架 | 转待维护、标签、交接 | 直接生效 |

### 9.2 各 Tab 下的批量操作

| Tab | 批量操作 | 说明 |
|-----|---------|------|
| 全部 | 批量提交上架、批量修改目录、批量交接 | - |
| 待维护 | 批量提交上架、批量不上架、批量修改目录、批量交接 | 提交上架需审批 |
| 待审核 | 批量通过、批量驳回 | 仅审批人可操作 |
| 已上架 | 批量下架、批量交接 | 直接生效 |
| 不上架 | 批量转待维护、批量交接 | 直接生效 |
