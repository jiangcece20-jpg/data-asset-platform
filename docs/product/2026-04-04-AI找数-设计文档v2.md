# AI 找数 — 设计文档 v2

- 文档版本：V2.0
- 文档日期：2026-04-04
- 文档状态：草稿，待确认
- 适用范围：数据资产管理平台 — AI 找数功能（重新设计）

---

## 1. 为什么重新设计

v1 方案存在以下根本性问题：

**假设错了**：v1 把"多轮对话 + 追问"作为核心体验，本质上是把用户变成了"被 AI 问问题的人"。但业务用户的真实心智是"我说一句话，你给我答案"，不想被反复追问。

**目标用户不聚焦**：v1 试图同时覆盖懂数仓的技术用户和不懂数仓的业务用户，导致设计上两头不讨好。

**入口形态不匹配**：弹窗适合"顺手一问"，但用户来找数通常是"专门来的"，弹窗空间局促，不适合浏览和对比结果。

---

## 2. 重新定义目标用户

### 2.1 核心用户（本期重点）

**业务用户** — 运营、产品经理、管理层

典型特征：
- 不懂数仓分层逻辑（不知道 ODS/DWD/DWS/ADS 的区别）
- 可能知道业务库的表名或字段名，但不知道数仓里对应的是什么
- 最终目标是"看到数据"，而不是"找到表"
- SQL 能力弱，但有时候愿意在辅助下自己查

典型诉求：
- "我想看各渠道的 GMV 表现" → 直接给我报表/看板
- "我们业务库有张 order 表，数仓里对应的是哪张？" → 帮我找到映射关系
- "我想看用户留存率，有没有现成的？没有的话我想自己查" → 先找报表，找不到给表 + 辅助写 SQL

### 2.2 次要用户（本期不重点，但不排斥）

**懂数仓的用户** — 数据分析师、数仓开发

他们的痛点是"搜出来太多，不知道哪个对"，这更像是搜索质量问题。AI 找数的语义理解能力对他们也有帮助，但不是本期设计的核心出发点。

---

## 3. 核心设计原则

### 原则一：结果优先，追问是例外

用户说一句话，AI 应该尽力给出结果，而不是先追问。
- 有把握 → 直接给结果
- 没把握 → 先给最可能的结果，同时提供"不是这个？试试这些方向"的引导
- 只有在完全无法推断时才追问，且追问必须给选项，不让用户自己想

### 原则二：报表/看板优先，表是降级选项

业务用户最终想"看到数据"，不是"找到表"。
- 优先推荐报表、看板（可以直接看）
- 找不到报表时，推荐表，并说明"这张表可以帮你看 XX，你可以这样查"
- 不要把底层 ODS 表推给业务用户

### 原则三：解释要用业务语言，不用技术语言

推荐结果的说明要用用户能听懂的话：
- ❌ "该表位于 DWS 层，聚合粒度为天级"
- ✅ "这是一张按天汇总的订单报表，包含各渠道的 GMV、订单量和退款率"

## 4. 功能范围

### 4.1 本期做

| 功能 | 说明 |
|---|---|
| AI 找数独立页面 | 专属页面，不是弹窗 |
| 自然语言 → 资源推荐 | 理解业务语言，优先推荐报表/看板 |
| 业务库表名/字段名映射 | 输入业务库表名，找到数仓对应的表 |
| 推荐结果卡片 | 展示资源类型、名称、业务说明、匹配原因 |
| 结果降级逻辑 | 有报表给报表，没有给表，并说明怎么用 |
| SQL 辅助（轻量） | 找到表后，可以描述需求让 AI 辅助生成 SQL |
| 追问引导 | 结果不准时，提供方向选项帮用户缩小范围 |

### 4.2 本期不做

| 功能 | 原因 |
|---|---|
| 指标值直接查询（"上月 GMV 是多少"） | 超出找数范围，属于数据查询产品 |
| 对话历史管理（搜索、删除） | 优先级低，先做好找数本身 |
| 推荐反馈（点赞/点踩） | 需要积累数据量才有意义，后续迭代 |
| 血缘关联推荐 | 复杂度高，后续迭代 |
| 真实 LLM + 向量检索后端 | 原型阶段先 mock |

### 5.4 业务库表名映射场景

用户输入业务库表名时，AI 通过**血缘关系**识别数仓中的下游表，并给出映射结果。

**实现依据**：平台已有血缘追溯能力，可以从业务库表（ODS 层来源）向下游追踪到 DWD/DWS/ADS 层的加工表。AI 找数利用这条血缘链，把"业务库表名 → 数仓表"的映射变成可查询的能力。

**输入示例**：`order` 表在数仓里是哪张？

**AI 回复**：
```
根据血缘关系，业务库 order 表在数仓中有以下下游表：

[表] dwd_order_detail — 订单明细表（DWD 层）
  直接来源于 order 表，保留了完整订单字段，
  额外补充了渠道、用户画像等维度

[表] dws_order_day — 订单日汇总表（DWS 层）
  基于 dwd_order_detail 聚合，按天汇总，适合看趋势

你想要的是明细数据还是汇总数据？
```

**字段级映射**：如果用户输入的是字段名（如 `order_id`），同样可以通过血缘关系找到包含该字段的数仓表，并说明字段在数仓中的对应名称（如有改名）。

### 5.5 SQL 辅助（次要功能）

在找到表之后，用户可以点击"辅助写 SQL"，进入轻量的 SQL 辅助对话模式：

- 用户描述需求："我想看最近 7 天各渠道的 GMV，按 GMV 从高到低排"
- AI 基于已找到的表，生成 SQL
- 用户可以继续追问调整（"再加上环比"、"只看华东区"），AI 迭代修改 SQL
- 满意后点击"去工作台执行"，携带 SQL 跳转到即席查询工作台

**为什么支持多轮追问**：SQL 需求往往需要多次调整才能准确，业务用户 SQL 能力弱，更依赖对话式迭代来逼近想要的结果。这里的多轮追问是有明确上下文的（围绕同一张表、同一个 SQL），不同于找数阶段的模糊追问。

**边界**：SQL 辅助是找到表之后的下一步，不是独立功能。入口只在表类型的推荐卡片上出现。

## 6. 推荐策略

### 6.1 结果优先级

同一个需求，按以下优先级推荐：

1. **报表**（BI 报表，可直接查看）
2. **看板**（Dashboard，可直接查看）
3. **指标口径**（有明确定义的指标，可以找到对应报表）
4. **DWS/ADS 层表**（已聚合，业务用户相对容易理解）
5. **DWD 层表**（明细表，需要一定 SQL 能力）
6. **API**（需要开发接入，业务用户一般不需要）
7. **ODS 层表 / 视图 / 标签**（不推给业务用户）

### 6.2 角色感知（简化版）

v2 不再维护六种细分角色，改为两类简单判断：

| 判断 | 依据 | 推荐策略 |
|---|---|---|
| 业务用户 | 部门（非技术/数据部）或职位关键词匹配（运营/产品/管理） | 报表 > 看板 > DWS 表，不推 ODS |
| 技术/数据用户 | 部门（技术/数据部）或职位关键词匹配（开发/分析/数仓） | 全类型，按相关度排序 |

**职位数据的不确定性处理**：用户中心的职位字段可能千奇百怪，无法覆盖所有情况。处理策略：
- 优先用部门判断（部门字段相对规范）
- 部门无法判断时，用职位关键词做模糊匹配
- 两者都无法判断时，**默认按业务用户策略**（宁可保守，不把底层表推给不懂的人）

### 6.3 追问策略（简化版）

| 情况 | AI 行为 |
|---|---|
| 有明确结果（1-5 个高相关度资源） | 直接给结果，不追问 |
| 结果较多（5 个以上相关度接近） | 给 Top 3-5，底部附方向引导标签 |
| 完全无法推断（输入过于模糊） | 给 2-4 个方向选项让用户选，选完直接给结果 |

**约束**：最多追问 1 次，超过 1 次强制给出当前最佳结果。

## 7. 关键交互细节

### 7.1 输入框

- 支持自然语言输入，也支持直接输入表名/字段名
- placeholder 给出具体示例，降低用户的输入门槛
- 支持回车发送

### 7.2 推荐卡片

每张卡片展示：
- 资源类型标签（报表/看板/表/API，用颜色区分）
- 资源名称（中文名为主，英文名为辅）
- 业务说明（用业务语言描述这个资源能帮你看什么）
- 来源系统 + 负责人
- 匹配原因（一句话，说明为什么推荐这个）
- 操作按钮：查看详情 / 申请权限 / 辅助写 SQL（仅表类型）

### 7.3 权限状态

卡片上直接展示当前用户的权限状态：
- 已有权限 → 直接显示"查看"按钮
- 无权限 → 显示"申请权限"按钮
- 申请中 → 显示"申请中"状态

### 7.4 历史记录

- 页面左侧或顶部展示最近的搜索历史（最近 10 条）
- 点击历史记录可以重新加载对应的结果
- 可以清空历史

## 8. 与现有功能的关系

| 功能 | 关系 |
|---|---|
| 普通搜索（资源发现） | 并列入口，AI 找数是语义找数，普通搜索是精确搜索 |

| 资源详情页 | 推荐卡片点击"查看详情"跳转到统一详情页 |
| 权限申请流程 | 推荐卡片上直接触发权限申请，不需要先跳转详情页 |

## 10. 原型阶段 Mock 策略

原型阶段不接入真实后端，使用以下 mock 策略：

- 预置 20-30 条 mock 资源（覆盖报表、看板、DWS 表、DWD 表）
- 基于关键词匹配模拟 AI 推荐逻辑
- 预置 3-5 个典型 query 的完整 mock 结果（退款、GMV、用户留存、配送时效、转化漏斗）
- 提供隐藏的角色切换入口，演示业务用户 vs 技术用户的不同推荐结果
- SQL 辅助部分使用预置模板，根据用户描述做简单替换
