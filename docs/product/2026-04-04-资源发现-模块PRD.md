## 2. 模块定位与背景

### 2.1 一句话定位

**资源发现**是面向数据消费方的**统一浏览与筛选入口**：在同一列表中查看「已上架资产」与「尚未资产化的资源」，支持按范围、状态、类型、目录树、负责人等缩小结果，并进入统一详情页做理解与权限申请。

### 2.2 要解决的问题

- 用户不清楚对象是否已资产化，需要**同一入口**同时看到资产与资源。
- 仅依赖「资产目录」按业务域浏览，不足以覆盖**技术名 / 来源系统 / 物理集群**视角的查找。
- 需要在**不做治理操作**的前提下，暴露足够的**治理态标签**（如待维护、不上架），帮助用户判断可用性与联系人。

### 2.3 非目标（本模块不做）

- 上架、下架、审核、驳回、改目录、批量治理、标签后台维护、异常工单处理（均在**资源管理**或其他后台模块完成）。
- 替换「数据搜索」的全局关键词检索职责；资源发现以**结构化筛选 + 列表浏览**为主，与全局搜索互补。

## 3. 目标与成功标准

### 3.1 产品目标

- 用户能在 **3 步内**从进入资源发现到打开目标对象详情（含筛选）。
- 列表中能**一眼区分**资产 / 资源及对象类型（表、视图、API 等）。
- 筛选条件可组合使用，且**选中态清晰**（避免「点了没反应」的交互）。

### 3.2 成功指标（可后续量化）

- 从资源发现进入详情的占比、平均点击深度。
- 筛选使用率（范围 / 状态 / 类型 / 负责人）。
- 空查询下用户调整筛选后再次命中的比例（可用埋点或访谈替代）。

## 4. 用户与场景

### 4.1 主要用户

| 角色 | 典型诉求 |
|------|----------|
| 数据分析师 / 业务取数 | 按域、表名关键字、类型找表；申请权限 |
| 数据开发 | 确认某表是否已上架、目录归属、负责人 |
| 新平台用户 | 在资产目录之外，用「资源」视角浏览平台数据对象 |

### 4.2 典型场景（用户故事摘要）

1. **找未资产化表**：范围选「仅资源」，类型选「表」，结合搜索关键字与左侧物理树缩小范围。
2. **只看正式资产**：范围选「仅资产」，必要时按类型与负责人筛选。
3. **排除不可用资源**：状态选「已上架」或排除「不上架 / 待维护」（具体策略见 §6.2）。
4. **按数仓物理位置找**：展开左侧「物理结构」，点选集群 / 库等节点（仅对资源侧数据生效，规则见 §7）。

## 5. 信息架构与页面结构

### 5.1 整体布局

- **左侧**：物理结构树（快捷入口：全部、未归属 + 可展开树节点）。
- **右侧主区域**：
  - **筛选与操作条**（建议视觉上为一组「面板」，与下方列表区分）
    - 第一行：关键词搜索、负责人选择、「我的数据」快捷项、**排序**（升/降序 + 排序字段）、**标签筛选**入口。
    - 第二行：**范围**（全部 / 仅资产 / 仅资源）、**状态**（全部 / 待维护 / 已上架 / 不上架）、**类型**（全部 / 表 / 视图 / API / 报表 / 指标 / 标签）；类型项较多时支持**横向滚动**，避免挤压。
  - **结果统计**：展示当前筛选条件下的总条数。
  - **结果列表**：卡片式列表（与资产目录列表风格一致）。
  - **分页**：底部分页器，切换页码后保留当前筛选条件。

### 5.2 与相邻模块的关系

| 模块 | 关系 |
|------|------|
| 数据搜索 | 可从全局搜索跳转至资源发现或详情；资源发现内搜索为**本页列表过滤**（或对接同一检索服务时的子集，实现待定）。 |
| 资产目录 | 仅展示已上架资产；资源发现**范围可含资源**，定位更广。 |
| 资源管理 | 治理操作唯一入口；资源发现**仅展示状态，不提供治理按钮**。 |
| 统一详情页 | 点击卡片进入；`domain`（资产/资源）决定详情内部分动作展示，见总册。 |

## 6. 功能需求

### 6.1 关键词搜索（FR-DIS-01）

- **描述**：在资源发现页内，对当前统一数据集（资产 + 资源合并视图）按名称、描述等字段做文本匹配（具体字段以实现/检索服务为准）。
- **占位提示**：与原型一致，如「请输入资产名称/描述关键字」。
- **行为**：输入即触发列表刷新（可 debounce）；清空则恢复仅受其他筛选条件约束的结果。
- **验收**：修改关键字后列表与「共 N 条」同步变化；无结果时有明确空状态（见 §9）。

### 6.2 范围筛选（FR-DIS-02）

- **选项**：全部、仅资产、仅资源。
- **规则**：
  - 「资产」指已进入资产目录对外展示的对象（与数据模型中 `domain=asset` 或等价标记一致）。
  - 「资源」指未以资产形态上架或标记为资源侧浏览的对象（`domain=resource` 或等价）。
- **验收**：切换选项后列表仅含对应范围；选中态在 UI 上明确可辨。

### 6.3 状态筛选（FR-DIS-03）

- **选项**：全部、待维护、已上架、不上架（与平台资源状态枚举一致；若存在「待审核」「已驳回」等是否在本页暴露，由产品二期决定，本期以原型为准）。
- **规则**：在「范围」结果集上进一步按治理状态过滤；**资产**通常对应「已上架」态，其它状态多见于资源。
- **验收**：与后端状态字段映射一致，无错杀、无漏筛；与 §8.1 卡片展示一致。

### 6.4 类型筛选（FR-DIS-04）

- **选项**：全部、表、视图、API、报表、指标、标签。
- **规则**：与对象 `type` 字段枚举一致；「全部」不做类型限制。
- **验收**：类型控件与「范围 / 状态」**视觉同一套分段控件规范**，避免一种下划线 Tab、一种实心按钮造成认知分裂。

### 6.5 负责人筛选（FR-DIS-05）

- **描述**：按技术负责人（及后续若统一「主负责人」口径则一并说明）过滤。
- **交互**：下拉或可搜索选择器；支持清空；展示当前选中人名。
- **验收**：选择与清空后列表与条数正确；与「我的数据」组合时逻辑无矛盾（见 FR-DIS-06）。

### 6.6 「我的数据」快捷筛选（FR-DIS-06）

- **描述**：一键仅看当前登录用户作为技术或业务负责人（以实现定义为准）的对象。
- **验收**：开启后列表全部条目负责人包含当前用户；关闭后恢复其它条件。

### 6.7 排序（FR-DIS-07）

- **描述**：支持排序字段切换 + 升序/降序切换。
- **字段（本期）**：使用热度、创建时间、更新时间（与原型一致）；热度定义可与详情页「浏览/查询」统计口径对齐。
- **验收**：切换字段或顺序后列表顺序符合预期且稳定可复现。

### 6.8 标签筛选（FR-DIS-08）

- **描述**：打开标签筛选面板，按一个或多个标签过滤列表（原型可为占位通知，上线前需定义标签来源：人工 / 系统 / 混合）。
- **验收**：选择标签后列表过滤正确；未选标签时不缩小结果。

### 6.9 左侧物理结构树（FR-DIS-09）

- **节点**：「全部」「未归属」及按来源系统 / 集群 / 库等拆分的树（与元数据可得性一致）。
- **规则（逻辑）**：
  - 「未归属」：展示无物理路径或未归属目录的资源（资产可排除或按产品规则处理，与原型一致）。
  - 选中某物理节点：仅保留物理路径属于该节点及其子路径的资源；资产是否出现在树筛选下按产品规则（默认原型为**资源侧**）。
- **计数**：节点可展示子集数量（可选）；需与列表总数字段一致或可解释。

### 6.10 结果列表卡片（FR-DIS-10）

每张卡片至少包含：

- 类型图标 + **技术名**（表/视图支持「库.对象名」分行或分样式展示）。
- **复制**技术名操作。
- **类型标签**（表/视图/API…）。
- **资产 / 资源**身份标签。
- 至多展示 **N 个业务标签**（如 3 个），超出省略或 `+N`。
- **申请权限**入口（跳转权限申请流程，规则见总册；若当前对象不可申请需禁用或说明）。
- **来源**（平台/系统缩写规则与 `getPlatformShort` 类产品逻辑一致）。
- **目录**（业务目录或 `-`）。
- **近 30 天查询量 / 浏览量**（展示与 tooltip 口径与总册详情页一致）。
- **描述**摘要。
- **技术负责人、业务负责人**。
- **创建时间、更新时间**。

**点击卡片主体**：进入统一详情页，传入对象 id 与 domain（资产/资源）。

### 6.11 分页（FR-DIS-11）

- **描述**：固定每页条数（原型为 8，可配置）；切换页码保留所有筛选与排序。
- **验收**：末页不足一页时展示正确；筛选后页码重置为第 1 页。

## 7. 交互与体验要求

- **筛选选中态**：同一组内单选互斥；切换后**立即**刷新列表（或统一 Loading 策略）。
- **性能**：首屏与筛选响应在可接受范围内（具体 SLA 由技术方案补充）；大数据量时左侧树与列表采用懒加载或分页由技术定。
- **无障碍**：主要分段按钮具备可读的 `aria` 角色/标签（如实现采用 button 分段）。
- **一致性**：资源发现内部分段控件与平台其它筛选组件风格统一（圆角、间距、主色选中态）。

## 8. 数据与逻辑要点（供研发对齐）

### 8.1 统一发现记录模型（逻辑）

列表每一行对应「发现记录」，至少包含：

- `id`、`domain`（asset | resource）、`type`、`name`、`display/summary`、`catalog`、`tags[]`
- `tech_owner`、`biz_owner`、`created`、`updated`、`heat`（或拆分查询/浏览指标）
- `status`（治理状态，用于状态筛选）
- 物理路径推断所需字段：`source_type`、`source`、`source_system`、`name` 等（与树筛选一致）

### 8.2 筛选组合顺序（建议）

1. 合并资产与资源数据源 →  
2. 应用关键词、范围、类型、状态、负责人、我的数据、标签 →  
3. 应用左侧树条件 →  
4. 排序 →  
5. 分页切片  

若实现顺序不同，需在技术方案中说明等价性。

## 9. 空状态与提示

- **无数据**：文案说明可调整筛选、清空关键字或前往资源管理（仅引导，不提供治理操作）。
- **树节点无子项**：禁用或隐藏规则由 UI 规范补充。
- **标签筛选未实现**：原型占位提示需替换为真实面板或下线该按钮至二期（排期决策）。

## 10. 验收标准（模块级检查清单）

- [ ] 范围、状态、类型三组筛选**互斥选中态正确**，且与列表内容一致。
- [ ] 负责人、我的数据、排序、标签（若已接）与列表、总条数一致。
- [ ] 左侧树「全部 / 未归属 / 子节点」与列表过滤逻辑符合 §6.9。
- [ ] 卡片字段齐全，资产/资源区分清晰；点击进详情参数正确。
- [ ] 分页与筛选联动正确。
- [ ] 页面**不出现**上架、改目录、标签治理、审核等治理类主按钮。
- [ ] 与 `2026-03-28` 中关于列表必须展示字段的条款**无冲突**；若有差异以本文 + 变更说明为准更新总册。
