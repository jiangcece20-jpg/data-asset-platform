## 2. 背景与问题

当前统一详情页以表类资产为核心模型构建，默认结构包括字段结构、样例数据、分区信息、DDL 变更等模块。该模型适合表、视图类资产，但不适合指标、标签、API 等非表类对象。

当前主要问题如下：

- 指标、标签、API 直接复用表类详情模型，信息结构不匹配
- 指标和标签若首屏直接进入血缘页，不利于用户先理解“定义”和“口径/规则”
- API 的核心信息是参数与调用方式，而不是字段结构
- 血缘图谱中点击不同类型节点时，抽屉页签没有按类型切换，体验割裂
- 右侧固定信息栏仍偏表类字段，难以体现指标、标签、API 的核心识别信息

因此需要对指标、标签、API 的详情页与血缘抽屉做类型化调整，形成统一但分类型的信息架构。

---

## 3. 产品目标

### 3.1 业务目标

- 让不同资产类型使用更符合业务心智的详情模型
- 提高用户对指标、标签、API 的理解效率
- 降低非表类对象详情页中的无效信息占比
- 统一血缘内外的信息结构，增强类型识别与使用连贯性

### 3.2 用户目标

- 业务用户进入指标详情后，优先看到指标定义、口径和使用场景
- 业务用户进入标签详情后，优先看到标签规则、值域和适用场景
- 开发或数据用户进入 API 详情后，优先看到参数、返回信息与调用说明
- 用户在血缘图谱中点击任意节点时，抽屉页签与该类型保持一致

### 3.3 设计目标

- 统一详情框架，不同类型按配置切换
- 首个 Tab 承担“解释对象是什么”的职责
- 血缘作为第二层验证与追溯信息，不作为指标/标签默认首屏
- 右侧固定栏只展示跨 Tab 都需要反复查看的关键信息

## 6. 核心设计原则

### 6.1 首屏先定义，第二屏看血缘

- 表类对象的核心是结构，因此首屏保留字段结构
- 指标的核心是口径，因此首屏应为指标定义
- 标签的核心是规则，因此首屏应为标签定义
- API 的核心是请求与返回，因此首屏应为参数信息

### 6.2 血缘是验证层，不是默认解释层

- 血缘用于确认来源、上下游影响、关系追踪
- 对指标和标签来说，用户在未理解定义前直接看血缘，理解成本高
- 因此指标与标签默认不以血缘作为首个 Tab

### 6.3 抽屉页签必须和节点类型匹配

- 表/视图类节点保留字段信息
- API 类节点展示参数信息，不展示字段信息
- 指标、标签、报表类节点不展示字段页签

### 6.4 右侧固定栏只保留关键信息

- 固定栏展示“身份信息”和“高频判断信息”
- 不展示只在某个 Tab 中才有意义的细节信息
- 不展示表类专属字段到指标/标签/API 中

## 10. 交互细节

### 10.1 默认 Tab

- 表 / 视图默认进入字段结构
- 指标默认进入指标定义
- 标签默认进入标签定义
- API 默认进入参数信息
- 报表默认进入报表定义

### 10.2 右侧固定栏

- 血缘全屏页不展示右侧固定栏
- 详情页内切换 Tab 时右侧固定栏保持不变
- 固定栏内容随资产类型变化

### 10.3 抽屉页签切换

- 点击血缘图谱节点后，按节点类型生成抽屉页签
- 若当前激活页签不在新节点支持范围内，则自动切换到第一个页签
- 抽屉标题始终显示当前节点名称

### 10.4 详情页与血缘页一致性

- 详情页的资产类型判断逻辑与血缘抽屉共用
- 详情页首个 Tab 与血缘抽屉页签都按类型配置驱动
- 相同类型节点在不同页面的命名与字段定义保持一致

## 11. 验收标准

### 11.1 详情页

- 指标详情页默认首个 Tab 为指标定义
- 标签详情页默认首个 Tab 为标签定义
- API 详情页默认首个 Tab 为参数信息
- 不同类型右侧固定栏字段不同且符合各自定义

### 11.2 血缘抽屉

- 表/视图节点显示字段信息页签
- API 节点显示参数信息页签
- 指标/标签/报表节点不显示字段信息页签
- 抽屉页签切换后内容与类型一致

### 11.3 文案与结构

- 指标类页面不再默认以血缘关系作为首屏
- 标签类页面不再默认以血缘关系作为首屏
- API 页面不再出现“字段结构”命名
- 报表页面不再沿用表类字段结构命名

---

## 12. 风险与注意事项

- 当前为原型与 mock 数据阶段，字段内容为示意性信息，不代表最终真实口径
- 标签与指标的定义内容后续可能需要和实际平台字段做映射
- API 若未来接入真实接口文档，参数和返回字段需要进一步拆为请求参数 / 返回参数
- 若后续继续扩展更多资产类型，建议继续沿用配置驱动模式，而不是在页面中增加分散判断逻辑

---

## 13. 结论

本次调整的核心不是“为不同类型做不同皮肤”，而是重新定义不同资产类型的核心信息优先级：

- 表先看结构
- 指标先看口径
- 标签先看规则
- API 先看参数
- 报表先看定义

在此基础上，详情页、右侧固定栏和血缘抽屉形成一致的类型化信息结构，能显著降低用户理解成本，并为后续真实元数据接入与治理能力扩展打下基础。
