# 资源发现与资源管理 - 功能边界及信息架构

**文档范围说明：** 本文档仅界定 **资源发现** 与 **资源管理** 的定位、概念与二者边界。平台其它一级能力（资产检索、资产目录、统一详情页、血缘追溯、即席查询、我的、权限中心、资产分析等）的细则与验收以 **`2026-03-28-数据资产管理平台-资源发现与详情页治理PRD.md`（总册）** 及 **`docs/product/2026-04-04-*-模块PRD.md`** 为准；若与本文冲突，以总册与模块 PRD 及变更说明为准。

---

## 1. 目标与命名调整

### 1.1 命名优化

- 将原“资源目录”更名为“资源发现”

### 1.2 命名原因

- “资源目录”容易让用户理解为治理台账或后台目录管理
- 该页面实际承担的是统一查询入口，重点是帮助用户查找“还未成为资产的数据资源”
- “资源发现”更符合用户心智，强调“查找、筛选、理解、申请”

### 1.3 页面定位

#### 资源发现

- 面向普通数据使用者
- 提供资产与资源的统一查询入口
- 重点帮助用户发现非资产数据，但结果范围包含所有资源和已上架资产

#### 资源管理

- 面向数仓工程师
- 提供资源治理后台
- 承接上下架、标签、目录归属、待维护、异常处理等治理动作

## 2. 核心概念统一

### 2.1 资源与资产的关系

- 资源和资产本质上是同一对象的不同状态
- 资源采集进入平台后，默认处于“待维护”状态
- 经过补充信息、提交审核并审核通过后，资源上架到资产目录，此时对外称为“资产”
- 因此详情页可以共用同一套页面模型，只是在不同状态下展示不同动作和提示

### 2.2 当前建议的状态流转

- 待维护
  - 资源采集进入平台后的默认状态
- 待审核
  - 数仓工程师补充信息后发起上架申请
- 已上架
  - 管理员审核通过，资源成为资产
- 已驳回
  - 管理员驳回，允许重新补充信息并再次提交
- 不上架
  - 数仓工程师主动标记，不进入资产目录
- 已下架
  - 已上架资产被下架，后续支持恢复

### 2.3 当前角色边界

- 数仓工程师
  - 资源维护
  - 标签补充和校正
  - 发起上架
  - 发起不上架
  - 下架与恢复
- 管理员
  - 上架审核
  - 驳回处理
- 普通用户
  - 在“资源发现”中查询资源和资产
  - 查看详情
  - 后续可扩展为申请权限、建议资产化

## 3. 功能边界

### 3.1 资源发现的定位

资源发现是统一检索入口，不是治理后台。

它要解决的问题是：

- 用户不知道某份数据是否已经资产化
- 用户需要先找到可用数据，再决定是否使用
- 用户需要在一个入口中同时查到“资产”和“非资产资源”

### 3.2 资源发现应保留的功能

- 统一搜索
  - 同时搜索资产和资源
  - 支持技术名、中文名、别名、标签、负责人、来源系统
- 分类筛选
  - 库、表、视图、API、报表、指标
- 结果筛选
  - 全部
  - 仅资产
  - 仅资源
- 属性筛选
  - 来源平台
  - 来源系统
  - 标签
  - 最近更新时间
  - 是否可申请
  - 是否已归属目录
- 统一列表
  - 明确展示“当前是资产还是资源”
  - 展示状态、负责人、来源、更新时间、标签
- 统一详情入口
  - 点击后进入同一套详情页
  - 根据状态展示不同动作
- 推荐能力
  - 最近新增
  - 热门资源
  - 同域相关推荐

### 3.3 资源发现不应承载的功能

- 上架操作
- 上架审核
- 驳回处理
- 标签后台维护
- 目录归属修改
- 异常治理
- 批量治理动作

### 3.4 资源发现建议新增的功能

- 资产/资源显式切换
  - 用于解决“用户需要同时查到资源和资产”的问题
- 结果卡片状态标识
  - 资产：已上架
  - 资源：待维护 / 待审核 / 已驳回 / 不上架 / 已下架
- 详情页中的下一步动作
  - 若为资产：查看详情、申请权限、收藏
  - 若为资源：查看详情、联系负责人、后续可扩展“建议资产化”
- 搜索结果排序
  - 相关度
  - 最近更新
  - 使用热度
- 资源质量提示
  - 缺负责人
  - 描述不足
  - 最近未更新

### 3.5 资源管理的定位

资源管理是资源治理后台，不是用户查询入口。

它要解决的问题是：

- 资源采集后如何补齐信息
- 哪些资源应该上架成为资产
- 哪些资源不应上架
- 上架后如何维护、下架、恢复
- 标签和目录归属如何规范管理

### 3.6 资源管理应承载的功能

- 上下架管理
- 标签管理
- 目录归属管理
- 待维护管理
- 异常治理
- 操作记录与审计

## 4. 资源管理模块拆解

### 4.1 工作台首页

目标：

- 帮数仓工程师快速看到当前待办

建议模块：

- 待维护资源数
- 待审核申请数
- 已驳回待修改数
- 已下架资源数
- 异常待处理数
- 我的待办

### 4.2 上下架管理

目标：

- 管理资源从“资源”到“资产”的状态变化

建议子模块：

- 待维护
  - 新采集资源
  - 缺必要信息资源
- 待审核
  - 已提交上架申请，等待管理员审核
- 已驳回
  - 驳回原因查看
  - 修改后重新提交
- 已上架
  - 查看已成为资产的资源
  - 支持下架
- 不上架
  - 主动标记为不纳入资产目录的资源
- 已下架
  - 支持恢复

建议操作：

- 补充信息
- 提交上架
- 重新提交
- 标记不上架
- 下架
- 恢复

### 4.3 标签管理

目标：

- 管理资源标签体系与资源打标质量

建议子模块：

- 标签字典
  - 系统标签
  - 人工标签
- 资源打标
  - 单个资源编辑标签
  - 批量打标
- 标签校验
  - 缺失标签
  - 重复标签
  - 不规范标签

当前建议规则：

- 系统自动生成少量基础标签
- 数仓工程师负责人工补充与校正
- 业务方是否参与标签建议，暂缓决策

### 4.4 目录归属管理

目标：

- 解决资源如何被用户更容易找到的问题

建议子模块：

- 未归属资源
- 待调整目录
- 批量归属
- 目录映射规则

建议方向：

- 当前“数仓视角目录”不利于业务查找，需要重构
- 目录不建议纯技术分层
- 更建议采用“业务域 + 主题域 + 数据对象”的结构

### 4.5 资源维护

目标：

- 让新采集资源尽快达到可治理、可上架状态

建议子模块：

- 缺负责人
- 缺中文名/描述
- 缺标签
- 缺目录归属
- 字段信息不完整
- 最近元数据未同步

### 4.6 异常处理

目标：

- 处理资源接入和治理过程中的异常

建议子模块：

- 元数据采集异常
- 字段结构变更异常
- 权限异常
- 质量异常
- 血缘异常

### 4.7 审计与记录

目标：

- 追踪谁对资源做了什么治理动作

建议子模块：

- 上下架记录
- 标签修改记录
- 目录归属修改记录
- 驳回与恢复记录

### 5.4 详情页信息架构

因为资源和资产是同一对象的不同状态，建议详情页统一：

- 基础信息
- 字段结构
- 血缘关系
- 管理信息

根据状态展示不同按钮：

- 待维护：补充信息、补标签、调整目录
- 待审核：查看申请、撤回或等待审核
- 已驳回：查看原因、重新提交
- 已上架：查看资产信息、下架
- 不上架：查看原因、恢复治理
- 已下架：恢复

## 6. 需要继续补充的关键决策

### 6.1 不上架是否需要审核

当前不清楚：

- 数仓工程师是否可以直接将资源标记为“不上架”
- 还是也需要管理员审核

建议：

- 短期可先不审核，直接标记
- 但需要记录原因和操作日志

### 6.2 是否支持“建议资产化”

当前不清楚：

- 普通用户是否可以在详情页发起建议资产化

建议：

- 可以保留入口，但先不做正式审核流
- 先作为“反馈给数仓工程师”的轻量入口

### 6.3 目录归属是否支持多目录

当前不清楚：

- 一个资源是否只允许归属一个目录

建议：

- 初期先单目录，降低治理复杂度
- 后续如存在明显跨域复用需求，再扩展为“主目录 + 辅助标签”

### 6.4 目录体系如何重构

当前你已经明确：

- 纯数仓视角目录不适合业务找数

建议优先方向：

- 一级：业务域
- 二级：主题域
- 三级：数据对象/场景

例如：

- 交易域 / 订单 / 订单履约
- 用户域 / 行为 / 用户访问
- 供应链 / 库存 / 库存变化

相比“ODS/DWD/ADS”这种技术分层，更适合业务用户查找。

## 7. 本轮建议落地结论

- 原“资源目录”更名为“资源发现”
- 资源发现承接统一查询入口，查资产也查资源，但更偏向发现非资产资源
- 资源管理承接治理后台，负责上下架、标签、目录、维护和异常
- 资源与资产共用详情页
- 当前最值得优先补的产品决策是：
  - 不上架是否审核
  - 目录体系重构方式
  - 是否支持建议资产化
  - 目录是否单归属
