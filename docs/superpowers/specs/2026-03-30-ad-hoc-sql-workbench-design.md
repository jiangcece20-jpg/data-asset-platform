# 即席查询工作台设计文档

## 背景

当前原型中已经存在一个 `SQL 查询` 页面和一个 `无权限 SQL` 页面，但它们主要服务于“从资源详情页进入单表查询”的场景，还不是面向用户的一级功能入口。

随着平台逐步形成“找数据 -> 看详情 -> 申请权限 -> 查询分析”的完整链路，需要新增一个独立一级菜单，用于承接用户对已有权限数据表的在线 SQL 查询需求。

本次设计目标是将现有 `SQL 查询页` 升级为一个独立的“即席查询工作台”，并放入顶部一级菜单，位置在“我的”之前。

## 目标

- 新增一级菜单 `即席查询`
- 支持用户在全局工作台内查询自己已有权限的表
- 支持多表 SQL 与 `JOIN`
- 支持在线查看结果
- 支持保存 SQL 草稿与查看历史查询
- 保持与现有详情页“去查询”链路兼容

## 非目标

- 不支持导出结果
- 不支持分享查询
- 不支持越权查询
- 不支持自动生成 SQL
- 不支持复杂开发者 IDE 能力，如多编辑器标签页、版本比对、脚本管理

## 用户与场景

### 主要用户

- 业务分析人员
- 数据使用者
- 对表有查询权限的业务角色与技术角色

### 典型场景

1. 用户进入平台后，直接打开 `即席查询`，搜索自己有权限的表并完成多表关联分析
2. 用户先在资源详情页确认字段和口径，再进入 `即席查询` 进行分析
3. 用户把常用 SQL 保存为草稿，后续继续补写和执行
4. 用户查看历史查询，复用曾经执行成功的 SQL

## 信息架构

顶部一级导航调整为：

`资产检索 / 资产目录 / 资源发现 / 资源管理 / 即席查询 / 我的 / 血缘追溯`

### 页面分工

- `资产检索 / 资产目录 / 资源发现`
  负责找表、看信息、申请权限
- `即席查询`
  负责对已有权限表进行 SQL 查询与结果预览
- `我的`
  负责承接草稿、申请记录、权限记录、收藏等个人视图
- `详情页`
  保留“去查询”的快捷入口，但不再承担主查询工作台职责

## 产品定位

即席查询是一个“轻量 SQL 工作台”，重点是降低业务用户对已授权数据的查询门槛，而不是打造一套专业开发 IDE。

设计原则：

- 围绕“已有权限表”组织查询，而不是围绕空白编辑器
- 以快速分析为主，而不是复杂开发
- 强化辅助能力，但不替用户做 SQL 决策
- 权限校验前置，避免越权执行

## 页面结构设计

页面采用三栏布局，中间区域为主工作区。

### 顶部工具栏

包含：

- 页面标题：`即席查询`
- 说明文案：`支持查询您已有权限的数据表，可进行多表关联分析`
- 当前查询状态提示
- 操作按钮：`新建查询`、`保存草稿`、`历史查询`

### 左侧：权限内数据表面板

职责是帮助用户快速定位可查询表，而不是展示全平台所有资源。

建议内容：

- 搜索框：支持按中文名、英文名、库表名搜索
- 筛选项：来源系统、资源类型、最近使用
- 分组区：
  - 最近使用
  - 我有权限的表
  - 收藏常用表
- 表项字段：
  - 中文名称
  - 灰色小字 `库名.表名`
  - 来源信息
  - 收藏状态

点击表项后：

- 进入当前查询引用表区域
- 右侧展示字段结构
- 支持插入表名到 SQL

### 中间：SQL 编辑与结果区

中间主区由三部分构成：

- 当前引用表标签区
- SQL 编辑器
- 结果区

工具能力：

- 执行
- 格式化
- 清空
- 保存草稿

结果区展示：

- 执行状态
- 执行耗时
- 返回条数
- 结果表格预览

### 右侧：字段辅助区

右侧提供当前选中表的字段信息，帮助用户补齐查询。

建议内容：

- 当前聚焦表
- 字段搜索
- 字段列表
- 每个字段展示：
  - 字段名
  - 字段类型
  - 字段注释

交互：

- 点击字段可插入 SQL 编辑器
- 当引用多张表时，可切换查看不同表字段

## 核心功能设计

### 1. 全局进入即席查询

用户点击一级菜单 `即席查询` 后进入空白工作台。

默认状态：

- 左侧展示已有权限表
- 中间展示新建查询引导
- 右侧为空或提示先选择数据表

适用于“我来直接分析数据”的用户心智。

### 2. 从详情页进入即席查询

详情页保留“即席查询”快捷入口。

交互效果：

- 跳转到同一个即席查询工作台
- 自动把当前表加入引用表区域
- 右侧自动加载该表字段
- 编辑器自动带入示例 SQL

如果当前用户无查询权限，则不进入工作台，而是进入无权限拦截态。

### 3. 多表 SQL 与 JOIN

支持用户在同一工作台中选择多张已有权限表，并手写 SQL 完成 `JOIN`。

系统提供的辅助包括：

- 插入表名
- 插入字段
- 展示当前已引用表

系统不负责自动拼装 `JOIN` 逻辑。

### 4. SQL 草稿

草稿用于保存正在编写但未必执行的 SQL。

保存内容建议包括：

- 草稿名称
- SQL 正文
- 涉及表
- 最近保存时间

交互规则：

- 首次保存时输入草稿名称
- 再次保存时默认覆盖
- 有未保存修改时，离开前提示是否保存

### 5. 历史查询

历史查询保存的是已经执行过的查询记录，用于后续复用。

展示字段建议包括：

- 查询名称或 SQL 摘要
- 执行时间
- 执行状态
- 涉及表

点击历史记录后：

- 回填 SQL 到当前编辑器
- 不强制恢复旧结果
- 用户可再次执行查看新结果

## 页面状态设计

建议原型覆盖以下状态：

1. 空白态
2. 已选一张表
3. 已选多张表
4. 查询成功有结果
5. 查询成功无结果
6. SQL 语法错误
7. 执行失败
8. 无权限拦截
9. 草稿列表态
10. 历史查询列表态

## 权限规则

### 基本规则

- 即席查询只展示当前用户已有权限的表
- 用户不能执行涉及无权限表的 SQL
- 用户即使手写了无权限表名，也不能绕过权限校验
- 草稿与历史查询仅本人可见
- 结果只支持在线查看，不支持导出

### 再执行校验

若用户打开了旧草稿或旧历史查询，执行前仍需重新校验涉及表权限。

若某张表权限已过期：

- 草稿内容仍可查看
- 执行时拦截并提示去申请权限

## 异常与提示设计

首期建议覆盖 3 类关键异常：

### 1. 无权限

提示示例：

`当前 SQL 引用了您暂无权限的表：xxx，暂不可执行`

附动作：

- 去申请权限

### 2. SQL 语法错误

提示示例：

`SQL 语法校验失败，请检查关键字、表名或字段名`

### 3. 执行失败

提示示例：

`查询执行失败，请稍后重试`

异常发生时：

- 不清空编辑器
- 保留用户当前 SQL
- 保留最近一次结果区状态，避免用户丢失上下文

## 与现有原型的衔接方案

本次不建议新建一套独立查询原型，而是升级现有 `page-sql`。

### 原型调整建议

1. 顶部导航增加 `即席查询`
2. 现有 `page-sql` 升级为全局工作台
3. 保留 `page-sql-noperm` 作为无权限拦截态
4. 保留从详情页进入即席查询的快捷路径

### 数据与 mock 建议

原型中建议补充：

- 已有权限表列表
- 最近使用表
- 收藏常用表
- SQL 草稿列表
- 历史查询列表
- SQL 执行校验结果

## 推荐实施方案

推荐采用“全局工作台 + 详情页快捷进入”的组合方案：

- 顶部新增独立入口，满足高频分析用户
- 详情页保留快捷进入，满足找表后立即查询的场景
- 工作台本身保持轻量，不做过重 IDE 化能力

## 结论

本次即席查询能力应作为平台一级功能建设，而不是详情页附属页面。

它的价值在于把“已有权限数据的在线分析”从详情页链路中独立出来，形成一个清晰、可复用、可沉淀草稿和历史的查询工作台。

在不增加复杂开发工具负担的前提下，这个设计能够较好承接业务用户的多表关联查询需求，并与现有资产平台结构保持一致。
