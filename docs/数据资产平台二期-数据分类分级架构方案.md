## 2. 功能定位

本功能为数据资产管理平台新增"权限申请与审批"能力，核心解决：

- 用户申请数据资产使用权限时，审批流能根据资产属性自动路由到正确的审批人
- 审批动作在飞书中完成，平台负责申请管理、路由匹配、状态同步
- 支持"申请单"模式批量积累资产，提交前预览拆单结果，后台自动拆单

**一句话定位：** 平台管申请和路由，飞书管审批，用户只管加入申请单和提交。

## 4. 核心功能模块划分

### 模块 1：权限申请（用户侧）— 申请单模式

**核心交互模型：** 加入申请单（跨页面积累）→ 申请单明细页（预览拆单 + 填理由）→ 提交

**入口 A：详情页**
- "申请权限"按钮：直接弹窗填理由并提交（单个快捷申请，不经过申请单）
- "加入申请单"按钮：加入申请单，Header 角标 +1，用户继续浏览

**入口 B：资源发现页 / 资产目录页**
- 每个列表项右侧增加"加入申请单"图标按钮
- 点击后该资产加入申请单，按钮变为"已加入"（置灰）
- Header 申请单角标 +1

**入口 C：Header 申请单图标**
- 申请单为空时：图标隐藏，不占位
- 申请单有内容时：图标显示在 Header 右上角（用户头像左侧），角标显示资产数量
- 点击进入申请单明细页

**申请单数据持久化：**
- 申请单数据持久化到后端（关联用户 ID）
- 用户关闭浏览器后重新打开，申请单内容仍在
- 提交成功后自动清空申请单

**加入申请单时的拦截规则：**

| 情况 | 处理 |
|---|---|
| 已有该资产权限 | 提示"你已拥有该资产权限"，不允许加入 |
| 该资产已在申请中 | 提示"该资产已有进行中的申请"，不允许加入 |
| 该资产已在申请单中 | 提示"该资产已在申请单中"，不允许重复加入 |
| 申请单已满（上限 50） | 提示"申请单已满，请先提交当前申请"，不允许加入 |

**申请单明细页交互：**

1. 资产列表区
   - 展示所有已加入的资产，每项显示：名称、类型、业务线、来源平台
   - 每项支持移除（移除后实时重新计算拆单预览）
   - 空状态：展示"申请单为空，去资源发现页添加资产"

2. 拆单预览区
   - 平台实时对申请单中的资产执行审批路由匹配
   - 按"审批模板 + 审批人"分组展示
   - 每组显示：包含的资产列表、审批节点链、各节点审批人姓名
   - 若某资产匹配不到审批模板，标红提示"该资产暂无匹配的审批流，请联系管理员"

3. 申请理由区
   - 统一申请理由输入框（必填，最少 10 字）
   - 下方有"为各审批流单独填写理由"开关
   - 开关打开后：每个审批流分组下方出现独立理由输入框
   - 独立理由输入框默认预填统一理由，用户可修改
   - 提交时：有独立理由的审批流用独立理由，没有的用统一理由

4. 提交
   - 点击"提交申请"
   - 平台为每个子单创建飞书审批实例
   - 成功后清空申请单，跳转到"我的申请"详情页
   - 失败时提示具体错误，申请单内容保留

### 模块 2：审批路由引擎（后台）

**路由匹配逻辑：**

```
输入：资产的 [类型, 业务线, 来源平台, 业务负责人]
  ↓
匹配：审批模板库（按优先级从高到低）
  ↓
  优先级 1：类型 + 业务线 + 来源平台 + 业务负责人（精确匹配）
  优先级 2：类型 + 业务线 + 来源平台
  优先级 3：类型 + 业务线
  优先级 4：类型
  优先级 5：默认模板（兜底）
  ↓
输出：审批节点链 + 飞书审批定义 ID + 各节点审批人
```

**审批节点链示例：**

| 场景 | 节点 1 | 节点 2 | 飞书审批定义 |
|---|---|---|---|
| 数据表（通用） | 申请人上级（串行） | 表负责人（或签） | 两级串行审批 |
| 报表 + 金融 | 金融业务线审批人（串行） | — | 单级审批 |
| 报表 + 物流 | 物流业务线审批人（串行） | — | 单级审批 |
| API（通用） | API 负责人（串行） | — | 单级审批 |
| 默认兜底 | 资产负责人（串行） | — | 单级审批 |

### 模块 3：批量拆单引擎（后台）

**拆单规则：**
- 用户提交 1 次批量申请 → 生成 1 个主申请单
- 平台对每个资产执行审批路由匹配
- 按"匹配到的审批模板"分组，相同模板 + 相同审批人的资产合并为 1 个子单
- 每个子单独立创建 1 个飞书审批实例

**拆单示例：**

```
用户选了 5 个资产：
├── 表A（数据表，通用）     ──→ 子单1：两级串行（上级→表A负责人）
├── 表B（数据表，通用）     ──→ 子单2：两级串行（上级→表B负责人）  ← 负责人不同，不合并
├── 报表C（报表，金融）     ──→ 子单3：单级（金融审批人）
├── 报表D（报表，金融）     ──→ 子单3：合并 ← 同模板同审批人，合并
└── API-E（API，通用）      ──→ 子单4：单级（API-E负责人）

结果：1 个主单 + 4 个子单 = 4 个飞书审批实例
```

### 模块 4：主子单状态管理

**子单状态机：**

```
待审批 → 审批中 → 已通过
                → 已驳回 → 重新申请 → 待审批（循环）
       → 已撤回
```

**主单状态机：**

```
审批中（所有子单都在审批中）
  → 部分通过（有子单通过，有子单仍在审批或已驳回）
  → 全部通过（所有子单都通过）
  → 全部驳回（所有子单都驳回）
  → 已撤回（用户主动撤回所有未完成子单）
```

**主单状态计算规则：**

| 子单状态组合 | 主单状态 |
|---|---|
| 全部待审批/审批中 | 审批中 |
| 有通过 + 有审批中或待审批 | 部分通过 |
| 有通过 + 有驳回 + 无审批中 | 部分通过 |
| 全部通过 | 全部通过 |
| 全部驳回 | 全部驳回 |
| 用户撤回 | 已撤回 |

**对接架构：**

```
平台                              飞书
  │                                │
  ├── 创建审批实例 ──────────────→ │ 审批定义（2-3个通用定义）
  │   POST /approval/instance      │
  │   - approval_code              │
  │   - form（申请理由、资产信息）  │
  │   - node_approver_list         │
  │     （动态指定各节点审批人）    │
  │                                │
  │                                ├── 审批人在飞书中操作
  │                                │
  │ ←──────────────── webhook ─────┤ 审批状态变更回调
  │   - instance_code              │
  │   - status（APPROVED/REJECTED）│
  │   - comment（驳回原因）        │
  │                                │
  ├── 更新子单状态                 │
  ├── 重算主单状态                 │
  └── 更新资产权限状态             │
```

**飞书审批定义规划（在飞书后台创建）：**

| 审批定义 | 节点结构 | 适用场景 |
|---|---|---|
| 数据权限申请-单级 | 1 个审批节点 | 报表、API、指标、默认兜底 |
| 数据权限申请-两级串行 | 2 个串行审批节点（第 2 节点或签） | 数据表 |

**表单字段（飞书审批表单）：**

| 字段 | 类型 | 说明 |
|---|---|---|
| 申请人 | 自动填充 | 发起人信息 |
| 申请理由 | 文本 | 用户填写 |
| 资产名称 | 文本 | 申请的资产列表 |
| 资产类型 | 文本 | 表/报表/API/指标 |
| 业务线 | 文本 | 所属业务线 |
| 来源平台 | 文本 | 资产来源 |

### 模块 6：审批模板管理（管理员侧）

**审批模板数据结构：**

| 字段 | 说明 |
|---|---|
| 模板名称 | 如"数据表-通用审批"，不可重复 |
| 匹配条件-资产类型 | 数据表 / 报表 / API / 指标 / 标签 / 全部（可选，不选=全部） |
| 匹配条件-业务线 | 金融 / 物流 / 用户增长 / ... / 全部（可选，不选=全部） |
| 匹配条件-来源平台 | Hive / MySQL / 报表平台 / ... / 全部（可选，不选=全部） |
| 匹配条件-目录 | 从目录树中选择，支持多选，选中父目录自动包含子目录（可选，不选=全部） |
| 审批模式 | 4 种预设模式（见下方） |
| 审批角色 | 仅当审批模式含"指定角色"时出现，从审批角色列表中选择 |
| 优先级 | 1-99，数字越小越优先，完全靠数字排序，管理员自己控制 |
| 状态 | 启用 / 停用 |

**匹配条件规则：**
- 所有条件都是可选的，但至少选择一项
- 不选 = 匹配全部
- 多个条件之间是 AND 关系
- 匹配时按优先级数字从小到大遍历，取第一个命中的模板
- 必须有一个兜底模板（所有条件都为"全部"），否则提示"存在未覆盖的资产"
- 不允许两个启用状态的模板有完全相同的匹配条件组合

**审批节点链（自定义配置）：**

管理员可自由添加多个审批节点，节点之间固定串行（按顺序依次审批）。

每个节点配置：

| 字段 | 类型 | 说明 |
|---|---|---|
| 审批人来源 | 下拉单选 | 申请人上级 / 资产负责人 / 指定角色 / 固定审批人 |
| 审批角色 | 下拉单选 | 仅当来源="指定角色"时出现，从审批角色列表选择 |
| 固定审批人 | 搜索选人 | 仅当来源="固定审批人"时出现，从 HR 系统搜索选择具体人员 |

规则：
- 至少 1 个节点，无上限（实际场景一般 1-3 个）
- 节点之间固定串行
- 只有 1 个节点时不允许删除
- 资产负责人多人时为或签，指定角色多人时为或签
- 固定审批人为单人审批

常见配置示例：

| 场景 | 节点链 |
|---|---|
| API 默认 | ① 资产负责人 |
| 报表按业务线 | ① 指定角色（金融业务线审批人） |
| 数据表通用 | ① 申请人上级 → ② 资产负责人 |
| 核心表严格审批 | ① 申请人上级 → ② 指定角色（业务线审批人）→ ③ 资产负责人 |
| 特殊审批 | ① 固定审批人（张三）→ ② 资产负责人 |

**审批角色管理：**

审批管理页面下设两个 Tab：审批模板 / 审批角色

| 字段 | 说明 |
|---|---|
| 角色名称 | 如"金融业务线审批人"，不可重复 |
| 成员 | 1-N 个人员，从飞书通讯录搜索选择，多人时审批为或签 |
| 被引用数 | 当前有多少个模板引用了该角色 |
| 操作 | 编辑 / 删除（被引用时不允许删除） |

### 模块 9：治理操作体系

#### 9.1 操作管控规则（不区分资产状态，统一适用）

| 操作 | 管控方式 | 入口 | 说明 |
|---|---|---|---|
| 改中文名/描述/使用说明 | 自助 + 记日志 | [编辑信息] 弹窗 | 信息补充，不影响结构 |
| 打标签 | 自助 + 记日志 | [编辑信息] 弹窗 | 只影响搜索筛选 |
| 补字段描述 | 自助 + 记日志 | 详情页字段 Tab 内编辑 | 纯补信息 |
| 变更目录 | 审批（数据管理员） | [变更目录] 独立弹窗 | 影响全公司找数路径，不论上没上架都影响 |
| 转交负责人 | 审批（转交人上级 → 被转交人 → 被转交人上级） | [转交负责人] 独立弹窗 | 影响审批流，需多方确认 |
| 上架 | 审批（数据管理员） | [提交上架] 按钮 | 资产对全公司可见 |
| 下架 | 审批（数据管理员） | [申请下架] 按钮 | 可能有人在用 |

#### 9.3 转交负责人审批流

**背景：** 负责人可以是多人（如 A、B）。转交时只转交操作人自己的负责人身份，不影响其他负责人。

示例：转交前负责人是 A、B，A 将自己的工作转给 C → 转交后负责人是 B、C。

**审批节点链（最多三级，有去重逻辑）：**

```
① 转交人的上级审批（确认可以转出）
  ↓
② 被转交人确认（确认愿意接收）
  ↓
③ 被转交人的上级审批（确认可以接收）
   — 若被转交人上级 = 转交人上级 → 跳过（已在①审批过）
   — 若被转交人上级 = 被转交人本人 → 跳过
```

**去重场景举例：**

| 场景 | 实际节点 |
|---|---|
| A 上级是 M，C 上级是 N（不同） | ① M → ② C → ③ N（三级） |
| A 上级是 M，C 上级也是 M | ① M → ② C（两级，③ 跳过） |
| A 转给自己上级 M | ① M 确认转出兼接收（合并）→ 跳过③（两级合一） |

**飞书审批定义：** 使用动态节点能力，通过 `node_approver_list` 动态传入 2-3 个节点，复用权限申请的多级串行审批定义。

**转交弹窗字段：**

| 字段 | 说明 |
|---|---|
| 当前负责人 | 只读，展示所有负责人 |
| 我的身份 | 只读，当前操作人 |
| 转交给 | 从 HR 系统搜索选人（单选） |
| 转交原因 | 必填 |

#### 9.4 飞书审批定义（更新）

共 4 个审批定义：

| 审批定义 | 用途 | 节点结构 |
|---|---|---|
| 权限申请-多级串行 | 权限申请（1-N 个节点） | 动态节点，通过 node_approver_list 传入 |
| 治理-上架/下架/目录变更 | 上架、下架、目录变更 | 单级，审批人=数据管理员 |
| 治理-转交负责人 | 转交负责人 | 动态节点（2-3 级），通过 node_approver_list 传入 |
| 治理-待我审批（被转交人确认） | 被转交人在飞书中确认接收 | 单级，审批人=被转交人 |

> 注：转交负责人的节点②（被转交人确认）和节点③（被转交人上级）合并在同一个飞书审批实例中，通过动态节点实现。

## 5. 关键交互流程

### 5.1 单个快捷申请流程（详情页直接提交）

```
用户在详情页点击"申请权限"
  → 弹出申请弹窗
  → 展示：资产名称、类型、业务线、来源平台
  → 展示：审批流预览（匹配到的审批节点链 + 审批人）
  → 用户填写：申请理由
  → 点击"提交申请"
  → 平台执行审批路由匹配
  → 调用飞书 API 创建审批实例
  → 详情页权限状态变为"申请中"
  → 审批人在飞书中收到审批通知
  → 审批人通过/驳回
  → 飞书 webhook 回调平台
  → 平台更新权限状态
  → 用户在"我的申请"中看到结果
```

### 5.2 申请单模式批量申请流程

```
用户在资源发现/资产目录/详情页浏览资产
  → 看到想申请的资产，点击"加入申请单"
  → Header 右上角申请单角标 +1
  → 用户继续浏览，继续添加更多资产
  → 点击 Header 申请单图标，进入申请单明细页
  → 查看已添加的资产列表（可移除）
  → 查看拆单预览（自动按审批模板分组，展示审批人）
  → 填写统一申请理由
  → （可选）展开为各审批流单独修改理由
  → 点击"提交申请"
  → 平台为每个子单创建飞书审批实例
  → 生成 1 个主申请单
  → 清空申请单，跳转到"我的申请"详情页
  → 各子单独立审批
  → 飞书 webhook 逐个回调
  → 平台逐个更新子单状态，重算主单状态
```

### 5.3 部分驳回后重新申请流程

```
用户在"我的申请"详情中看到某子单被驳回
  → 查看驳回原因
  → 点击"重新申请"
  → 弹出申请弹窗，预填原申请理由（可修改）
  → 用户修改申请理由后提交
  → 平台为该子单重新创建飞书审批实例
  → 子单状态回到"审批中"
  → 主单状态重新计算
```

## 8. 风险与注意事项

1. **飞书审批 API 能力边界**：需要先调研确认飞书是否支持动态指定各节点审批人、或签节点等能力，这直接影响审批模板的设计
2. **webhook 可靠性**：飞书 webhook 可能存在延迟或丢失，建议增加定时轮询兜底机制
3. **组织架构接口**：获取"申请人上级"依赖组织架构接口，需确认接口可用性和数据准确性
4. **资产业务线属性**：当前资产元数据中是否已有"业务线"字段？如果没有，需要先补充
5. **审批模板冲突**：多个模板可能匹配同一资产，优先级规则需要在管理界面中清晰展示
6. **飞书审批定义维护**：需要在飞书后台手动创建 2 个审批定义，后续如果审批结构变化需要同步维护

## 9. 总结

本方案的核心架构思路是"三层分离"：

- **用户层**：只关心"选资产 → 填理由 → 提交"，不感知审批路由和拆单逻辑
- **路由层**：根据资产属性自动匹配审批模板，生成审批节点链，处理拆单
- **审批层**：飞书负责审批流转，平台通过 API 创建、通过 webhook 接收结果

这样平台不需要自建审批引擎，利用飞书成熟的审批能力，平台专注于路由匹配和状态管理。
